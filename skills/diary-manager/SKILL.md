## 日期判定规则

**所有时间均以用户所在地时区为准，不是电脑系统时间。**

在 skill 的 `config.local.md` 中配置系统时区和用户时区，以及时差换算公式。

用户的一"天"是当日 07:00 ~ 次日 07:00（用户所在地时间）。

- 当前用户时间在 07:00~23:59 → 日记日期 = 今天
- 当前用户时间在 00:00~06:59 → 日记日期 = 昨天

**实际操作**：每次追加日记前，**必须重新执行**时区转换命令获取当前用户时间。不要复用对话中更早查过的时间，因为用户可能在跨天后才回来继续对话。先拿到准确时间，再判断日记日期和写入时间戳。

示例（以 PDT 系统时区 + 北京用户时区为例）：
  - 北京时间 5月10日 23:00 写日记 → 文件 = 2026-05-10.md
  - 北京时间 5月11日 02:00 写日记 → 文件 = 2026-05-10.md（还算前一天）
  - 北京时间 5月11日 06:59 写日记 → 文件 = 2026-05-10.md（还算前一天）
  - 北京时间 5月11日 07:00 写日记 → 文件 = 2026-05-11.md（新的一天开始）

## 天气信息

每次创建新日记时，在标题下方第一行写上日期、星期、天气，像手写日记一样：

```
**5月22日 星期五 小雨 17℃**
```

获取方式：
1. 用 `curl -s "https://wttr.in/{城市}?format=%C+%t&lang=zh"` 获取当天天气和温度（城市从 config.local.md 读取）
2. 用 python 计算星期几：`from datetime import datetime; ['星期一','星期二','星期三','星期四','星期五','星期六','星期日'][datetime(YYYY,MM,DD).weekday()]`
3. 日期用中文格式：`M月D日`（不补零）

## 绝对时间原则

**所有涉及时间的记录都必须使用绝对日期+时间（如"5/27 12:08"），禁止使用"今天""昨天""早上""过阵子"等模糊说法。**

适用范围：
- 日记写入时的时间戳：`## HH:MM` 前必须确认绝对日期
- 记账条目：记录和确认时都带绝对日期（如"5/27 午饭 ¥20"）
- TODO/提醒：创建时间、计划完成时间、截止日期一律写绝对值
- 对话中确认记录时：回复用户也要带绝对日期（如"已记到5/27，打车 ¥11.35"），方便用户当场发现日期错误

**对话中周期性时间确认**：当对话持续时间较长（超过1小时），或用户提到带时间线索的事件（"早上""中午""昨天"），主动重新执行 `TZ='{{用户时区}}' date` 确认当前日期，并与用户提到的时间做交叉验证。不要等到写入时才查——提前查可以避免整串错误。

## 写入流程

1. 用户说"记一下"/"记个日记"等触发词
2. **先检查当日日记是否已存在**：读取 `{{日记根目录}}/YYYY-MM/YYYY-MM-DD.md`
   - 已有：追加到文档末尾，用 `## HH:MM` 时间戳分隔
   - 没有：先创建 `YYYY-MM/` 目录（如需），再按 `templates/daily-entry.md` 模板新建，填入日期和时间
3. 文档格式：

```markdown
# {{用户称呼}}的日记 · YYYY-MM-DD

## HH:MM

用户说的内容（原样记录，不做修饰）

---

## HH:MM

（追加的内容）

---
```

## 智能提取规则

每次追加日记后，扫描内容并生成提取计划，列出给用户确认：

### 提取类别

1. **TODO/待办**
   - 触发词：要做、得做、计划、安排、别忘了、下周要、回头要
   - 格式：可执行的 TODO 项 + 来源日期

2. **产品灵感**
   - 触发词：可以做个、如果能、想法、灵感、产品、要不试试、小工具、小玩意
   - 格式：灵感标题 + 简述 + 来源日期
   - ⚠️ **灵感 vs 分享/观察的区分**：用户转述别人的成功案例（如"某独立开发者做产品出海月入$6-7K"）属于行业观察或分享内容，**不是产品灵感**。只有当用户自己表达了"我也想做""可以试试""如果能把X做成Y"等明确的构建意图时，才算灵感。别人做得好 ≠ 用户想做。不确定时问一句："这个是当灵感记还是当分享记录？"

3. **行业观察/商业判断**
   - 触发词：这个行业、竞品、市场、趋势、风向
   - 格式：观察点 + 判断 + 来源日期

### Tracker 同步与待办概览

1. **如果用户提到待办更新，同步到 todo tracker。** 用户在对话中经常会说"这个做完了""这个不用了"，必须即时更新 `todo.md`，不能只在聊天中确认。
2. **每日提炼推送中附加待办概览。** 读取 `tracker/todo.md`，统计活跃待办总数，列出近期到期或标⭐的项，附在提炼内容末尾：
   ```
   📋 待办概览：共X项活跃待办，近期需关注：[列出近期到期/标⭐的项]
   ```


### 提取确认流程

1. 追加日记后，列出提取计划：
   ```
   📋 提取计划：
   1. [TODO] xxx → 来源：YYYY-MM-DD 日记
   2. [灵感] xxx → 来源：YYYY-MM-DD 日记
   3. [观察] xxx → 来源：YYYY-MM-DD 日记
   4. [Tracker同步] culture.md：xxx进度更新 → 来源：YYYY-MM-DD 日记
   5. [Tracker同步] study.md：xxx进度更新 → 来源：YYYY-MM-DD 日记
   ```
   **第4步（Tracker交叉检查，必须执行）**：扫描本次日记内容是否涉及已有 tracker 条目的更新。逐项检查：
   - 日记提到学习/课程/考试进展 → 检查 culture.md + study.md 是否需要更新进度
   - 日记提到看书/看小说/看剧 → 检查 culture.md 是否需要更新状态
   - 日记提到身体/饮食/运动 → 检查 lifestyle.md 是否需要更新
   - 日记提到花钱 → 检查 spending.md 是否需要追加
   - 日记提到恋爱/择偶/相亲/亲密关系 → 检查 relationship.md 是否需要更新
   - 如果某条内容已在 tracker 中有条目但进度过时，必须列出同步项
   - **高频遗漏场景**：用户在日记中提到"看了几节XX课""XX课学到第N课"，culture.md 中的进度列常被忘记更新（已踩坑：吴军课程日记6/2记了看2节，culture.md仍写"未开始"，直到6/7用户发现）
2. 用户确认后才写入对应位置
3. 不确定的内容先问用户

### 每日提炼 Cron

- 调度：Cron 时间需按系统时区填写（从 config.local.md 读取 Job ID 和调度时间）
- 推送方式：微信或其他已连接渠道
- 工具集：file, terminal, session_search
- 关联技能：diary-manager

每天自动触发，提炼"上一个日记日"的内容：

1. 计算上一个日记日：当前时间在 07:00 之后，所以上一个日记日 = 昨天
2. 读取上一天日记日对应的文件
3. 如果昨天没有日记，发送提醒消息，然后结束
4. **扫描微信对话补充信息**：用 session_search 搜索目标日记日 07:00 ~ 次日 07:00 的微信渠道对话，提取用户提到的事件、完成的事、想法等。仅作为提炼的补充信息源，不修改日记原文。已在日记中的内容不重复。
5. **双通道输出**（先存本地，再推微信）：
   a. 综合日记原文 + 微信对话补充，生成提炼内容（自然段叙述风格）
   b. 将提炼写入日记文件**顶部**（替换占位符），**绝对不能追加到文件末尾**
   c. 生成微信推送版本（简洁友好，适合手机阅读），作为最终输出返回

参见 `templates/daily-entry.md`，包含标准结构：头部提炼区、时间戳日记区、待办区、记账区。

## 📅 周总结与月总结

### 周总结

- Cron Job ID 从 config.local.md 读取
- 推送方式：微信
- 流程：
  1. 读取上周所有日记文件（上周一 07:00 ~ 本周一 07:00）
  2. 汇总内容：日记要点回顾、情绪/状态变化趋势、产品灵感汇总、TODO 进展、本周总开支
  3. 代理视角：一段话总结本周 + 鼓励/建议
  4. 存入 `{{日记根目录}}/YYYY-MM/weekly-YYYY-MM-DD.md`（周一日期）
  5. 推送微信

### 月总结

- Cron Job ID 从 config.local.md 读取
- 推送方式：微信
- 流程：
  1. 读取上个月所有周总结文件（基于周总结汇总，不再重读日记原文）
  2. 汇总内容：月度要点、产品灵感及进展、TODO 完成情况、月度总开支、情绪/成长轨迹
  3. 代理视角：一段话总结本月 + 下月寄语
  4. 存入 `{{日记根目录}}/YYYY-MM/monthly-YYYY-MM.md`
  5. 推送微信

## 📊 生活主线追踪器（Tracker）

用户的日记内容是多条主线交织的碎片，需要在每日提炼时同步整理到独立的 tracker 文件中，方便按主线查看完整脉络。

### 文件结构

```
{{Tracker 目录}}/
├── spending.md     # 月度花销汇总 + 分类统计
├── study.md        # 考证/学习进度 + 达成率
├── lifestyle.md    # 养生目标、饮食记录、体重、运动
├── culture.md      # 书/小说/电影/艺术展
├── relationship.md # 亲密关系追踪（择偶/恋爱/自我认知/关系模式）
└── todo.md         # 活跃的目标和待办（完成后归档）
```

### 同步规则

详细的 tracker 文件结构、类别枚举见 `references/tracker-conventions.md`。

1. **spending.md**：每次记账时同步追加
2. **study.md**：学习/考试/求职进展时同步
3. **lifestyle.md**：身体/饮食/运动/养生时同步
4. **culture.md**：看展/读书/看电影时同步
5. **relationship.md**：恋爱/择偶/相亲/亲密关系自我分析时同步
6. **todo.md**：新待办或目标时同步；完成的移到已完成区

## 🍜 饮食卡路里记录

用户每天吃饭时会告诉吃了什么或发食物照片，需要同步记录食物内容和估算卡路里。

### 触发方式

- 用户直接说吃了什么（文字描述）
- 用户发食物照片（用视觉模型识别）

### 流程

1. **识别食物**：文字直接理解；照片用视觉模型识别菜品/食物种类和大概份量
2. **估算卡路里**：基于常见食物热量数据估算（单位：kcal）
3. **写入当日日记**：在日记中追加饮食记录，包含食物名称 + 估算卡路里
4. **同步到 lifestyle tracker**：追加到 `lifestyle.md` 饮食记录表，同时保留养生规则判定

### 卡路里估算参考

常见食物热量（单人份，kcal）：
- 米饭（一碗 200g）：~230
- 面条（一碗）：~300-400
- 饺子/蒸饺（10个）：~400-500
- 麦当劳套餐（汉堡+薯条+可乐）：~800-1000
- 肯德基套餐：~800-1000
- 酱大骨（一份）：~500
- 蔬菜沙拉（一份）：~150-250
- 三文鱼刺身（一份）：~200-300
- 排骨汤（一碗）：~250-350
- 酸奶/茉酸奶（一杯）：~200-300
- 自助餐：视内容 ~1500-2500
- 火锅（一人）：~1000-1500
- 碱水面包/碱水卷（一个）：~200
- 小笼包/小笼汤包（一个）：~50-60
- 小馄饨（一碗）：~200-250
- 鸡肉肠（一根）：~100-150
- 红树莓/莓果（一盒）：~40-60

### lifestyle.md 饮食记录格式

```markdown
| 日期 | 饮食内容 | 估算卡路里 | 符合养生规则 |
|------|----------|-----------|-------------|
| 06-03 | 午饭：米饭+宫保鸡丁 | ~550 | ⚠️ 辣 |
| 06-03 | 下午茶：拿铁 | ~150 | ⚠️ 下午4点前OK |
```

### 日记中的饮食记录格式

在日记追加时，饮食记录嵌入正文：

```
## 12:30

午饭：米饭 + 宫保鸡丁，估算约 550 kcal。
```

### 注意事项

- 卡路里是**估算值**，用 `~` 前缀表示（如 `~550`），不要精确到个位数
- 用户只说了食物名没说份量时，按标准单人份估算
- 照片识别可能不完整，识别后跟用户确认"看起来是XX，对吗？"
- 一天多次进食，分多行记录，在周总结时汇总当日总摄入
- 不评价"吃多了/吃少了"，只记录数据，周总结时再给建议
- ⚠️ **视觉 API 不可用时不要阻塞**：如果 vision_analyze 返回 401/超时/其他错误，直接用文字问用户相关信息（如"多少钱？""看起来是什么？"），不要卡在重试上。用户通常乐于直接告诉你。

### 每日提炼 Cron 与 Tracker

每日提炼 cron **只做日记提炼，不更新 tracker**。
Tracker 由代理在对话中记录日记时实时同步。
周总结 cron 读取日记 + tracker 出周报。
月总结 cron 读取周总结 + tracker 出月报。

## 💰 每日记账

日记中嵌入每日开支记录，作为日记的一部分，不单独建文件。

### 格式

在日记文档中维护记账区块：

```markdown
## 💰 今日开支

| 项目 | 金额 |
|------|------|
| 午饭 | ¥25 |
| 咖啡 | ¥18 |
| **合计** | **¥43** |
```

### 规则

1. **只记用户明确说"我花的钱"的项目。** 别人请客的不能算在用户头上。
2. **绝对不能根据上下文推断用户可能还花了什么钱。** 只记用户原话中逐条提到的具体金额。
3. 用户口头描述花销时要**逐条确认**，不要自作主张归类或添加。
4. 每笔记录在「金额」列右对齐，末行用加粗合计
5. 同一天多次追加时，合并到同一个表格，更新合计
6. 如果当天日记还没有记账区块，自动创建
7. 提取阶段不单独提取记账数据（留在日记原文即可）
8. **预充值账户的后续扣款不重复记账。** 如果用户一次性充值了一笔大额（如苹果账号 $500），后续从该余额中自动扣除的费用（如 Codex、Claude Code 订阅）只在日记中提及即可，不再记到开支表/tracker。充值本身已记录过，重复记会虚增开支。

### Pitfalls

- ❌ **不要存到临时目录或其他路径。** 日记根目录从 config.local.md 读取。必须用 write_file 写到正确路径，绝不能自创新目录。
- ❌ 不要用 write_file 写一个完全独立的文件——如果当天已有日记，必须追加（用 patch 或在末尾追加）
- ❌ 记账区块不要单独建文件，嵌在日记文档中即可
- ❌ **不要编造记账条目。** 只记用户原话中逐条提到的具体花销。不能根据上下文推测——没说的就是没有。
- ❌ **不要把别人付的钱算在用户头上。** 必须等用户明确说"我花了XX"才记。
- ❌ 不要忘记每段追加内容前加 `## HH:MM` 时间戳（用户所在地时间）
- ❌ **不要用电脑系统时间判断日记日期。** 必须先换算为用户所在地时间再判断。
- ❌ **不要在未显式计算目标日期的情况下打开"最近的"日记文件然后往里写。** 正确流程：①查用户时间 → ②算出目标日期 → ③读/写该日期对应的文件。**目标日期是算出来的，不是猜出来的。**
- ❌ **每次涉及日记记录操作时，都必须重新查用户所在地时间，即使对话中之前已查过。** 用户可能跨天后继续对话，之前查的时间已经过期。**无例外：每一条日记写入、记账、追加内容前，都必须先重新确认当前日期。**
- ⚠️ **patch 修改 markdown 表格时注意竖线格式。** 保持单个 `|` 分隔，不要写成 `||`。
- ⚠️ **patch 替换多行内容时注意换行符。** 替换时务必保留原始的空行和换行结构。
- ❌ **记账前必须检查前面几天的日记，避免重复记账。** 遇到跨天关联的开销（提前买的票、预售商品等），先搜索近期日记确认是否已记录。
- ❌ **Cron schedule 用电脑系统时区执行。** 设定 cron 时间时务必换算。
- ❌ 不要在第一次写入时跳过提取确认流程。追加完成后必须列出提取计划，等用户确认后才写入对应位置。
- ❌ **不要只在聊天渠道上回复用户，却不把内容写进日记文档。** 代理的回复也属于日记的一部分，必须同步写入 .md 文件。
- ❌ **日记里提到学习/文化消费进展时，忘记同步更新对应 tracker。** 写完日记后必须执行 tracker 交叉检查：日记提到的课程进度→更新 culture.md + study.md；小说/书籍状态→更新 culture.md；身体饮食→更新 lifestyle.md。已踩坑：吴军课程在日记和 todo.md 都有进度记录，但 culture.md 和 study.md 一直写"未开始"，直到用户翻 tracker 时才发现（2026-06-07）。
- ❌ **用户告诉你"某件事做完了""我搞定了XX"时，必须立刻写入日记，不能只回复"厉害了""辛苦了"就过了。** 典型场景：用户深夜说"两件事我都搞定了"，代理感叹效率高但没落盘，第二天提炼就丢失了这段信息。正确做法：确认内容后立即追加日记条目。
- ❌ **不要用列表式格式写日记提炼。** 必须用自然段叙述风格，像朋友写的一段话，末尾附关键词/情绪基调/提及事项/开支。
- ❌ **TODO 提取后必须落盘到文件**（`{{Tracker 目录}}/todo.md`），不能只在对话中列出。
- ⚠️ 用户说的时间是 24 小时制。结合"07:00~次日07:00 为一天"的规则判断日期归属。
- ⚠️ **Cron 时区问题**：Cron schedule 必须按系统时区填写。换算公式从 config.local.md 读取。
- ❌ **绝对不要脑补或编造记账条目。** 拿不准的逐条确认，不要自作主张。
- ❌ **创建提醒类 cron job 时，prompt 不要让 agent 主动调用 send_message。** cron 的 `deliver` 参数会自动将 agent 的最终输出投递到微信。prompt 应该写"直接输出提醒内容作为最终回复"，而不是"发一条微信消息提醒"。如果 prompt 里写了"发微信消息"，cron 里的 agent 会误以为自己没有微信能力而放弃任务（已踩坑 2026-05-25，job_id 21fe5572fff5）。
- ❌ 记账前先确认每笔的付款人。自费和他人付费混杂时，必须问清楚。
- ⚠️ **语音输入价格容易出错。** 用户通过语音报购物清单时，金额经常被语音转文字误识别（如"百五十二"→"52"、"一百一十二"→"12"）。遇到大批量购物清单时，逐条确认关键金额，不要急于全部写入。
- ⚠️ **视觉 API key 可能被截断。** config.yaml 中的 api_key 如果显示为 `sk-sp-...b721` 这种带 `...` 的形式，说明 key 被截断/脱敏了，不是完整 key。遇到 401 invalid_api_key 时，先用 `read_file` 检查 config.yaml 中 key 是否完整，不要只归咎于"服务到期"。发现截断后提醒用户重新复制完整 key。备用通道：智谱 MCP zai_vision。
- ⚠️ **退货/退款处理：** 当用户决定退货某件商品时，在 tracker spending.md 中将金额列改为 `~~¥XX.XX~~ 已退货`（删除线+标记），同时同步更新月度汇总和按周汇总的数字。在日记中追加退货决定说明。
- ⚠️ **充值类开支的后续扣款规则：** 如果用户明确表示某笔充值（如苹果账号、会员卡）已覆盖后续消费，后续从该账户扣款的费用只在日记中提及，不再重复记入开支表/tracker。
- ⚠️ **消费决策时适度劝省。** 当用户提到要买东西时，先问"真的需要吗？"。但尊重用户的最终决定。
- ⚠️ **不要给医疗/用药建议。** 可以分享常识性信息，但不要推荐药物组合或用量方案。
- ❌ **不要在未读取当日日记的情况下评论"今天/明天该做什么"。** 先确认哪些事已经发生了。
- ❌ **不要用电脑系统时间做任何时间判断。** 即使不是写日记，也必须先换算用户所在地时间。
- ❌ **不要根据"记忆中的时间"说"晚安"或"早点休息"。** 每次回复前如果涉及时间判断，必须先查当前用户时间。
- ❌ **禁止在北京时间 22:00 前催用户休息/说晚安/说"早点睡"。** 这是高频踩坑（已发生3次以上）。即使用户提到"凌晨1点睡""4点起床"等，只要当前北京时间不到22点，就不能说"去睡吧""早点休息""晚安"。用户提到作息是为了记录，不是为了被催。正确做法：先 `TZ='Asia/Shanghai' date` 确认当前时间，再决定是否合适说时间相关的话。下午4点半催人睡觉是非常离谱的错误。
- ❌ **跨天陷阱：对话跨过午夜后，用户提到"早上""中午"等时间词时，必须立即重新查用户所在地时间。** 用户凌晨1点说"早上打车11.35"，那个"早上"是今天的早上，不是昨天的。典型场景：用户从晚上聊到凌晨，提到"早上"做的事 → 这是新的一天，不是前一天。每次记账/追加日记前，先执行 `TZ='Asia/Shanghai' date` 确认当前日期，绝不能依赖对话开头查过的时间。已发生事故：5/26晚上21:00开始对话，凌晨后用户说"早上打车""中午鸡薯双全"，代理全写到5/26，实际应是5/27（2026-05-27 踩坑）。
- ❌ **确认记录时不带绝对日期，导致用户无法当场发现错误。** 回复用户时必须带绝对日期（如"已记到5/27"），不能只说"记上了"。如果本次事故中回复时带了"已记到5/26鸡薯双全¥20"，用户就能当场发现日期错了（2026-05-27 踩坑）。
- ❌ **记账前检查是否已在之前的日记中记录过。** 遇到"之前买的""提前订的"等措辞时，先搜索近期日记确认。
- ⚠️ **语音报账数字易出错（高频踩坑）。** 用户通过语音/语音转文字报一长串开销时，数字经常被截断或识别错误（如 ¥152→¥52、¥112→¥12）。**必须先完整列出每笔金额让用户确认，确认后再写入。** 不要先写再改，来回修很浪费时间。
- ⚠️ **预充值/余额类账目不重复记账。** 当用户一次性充值一笔大额到账户（如苹果账号 $500），后续从该余额扣除的费用（如 Codex、Claude Code 订阅）只在日记文字中提及即可，不重复记到开支表格和 tracker。避免重复计算。
- ❌ **Cron 提醒类任务：prompt 不要让 agent 主动调用 send_message 或说"发微信消息"。** 正确做法：prompt 直接写"输出提醒内容作为最终回复"，系统通过 `deliver: "weixin"` 自动投递到微信。如果 prompt 写了"发微信消息"，cron agent 会误以为自己没有微信发送能力而放弃任务（已踩坑 2026-05-25，job 21fe5572fff5）。工具集只需 file + terminal，不需要额外的消息发送工具。
- ⚠️ **处理多段录音时注意顺序标记。** 用户可能分多次发来长文本/ASR录音转写，容易搞混哪段是哪段。每次收到时先确认是第几段，整理时在文件中按实际顺序标注"第一段/第二段"，不要靠文件中的位置猜测。已发生事故：2026-06-07 处理青藤红娘录音时，把两段顺序搞反了（第二段内容更短被当成第一段），用户纠正后才修正。

## Cron 执行注意事项

### 模型限频问题（429）

当使用的模型存在限频时：
1. 确认是限频而非额度耗尽
2. 配置 fallback_providers 作为降级模型
3. 多个 cron 任务时间错开
4. 失败后手动补跑：`cronjob action=run job_id=<id>`

### 提炼内容写入位置与格式

云青的每日提炼写在日记文件的**顶部**（标题下方），用 `> 📝 云青的提炼：` 引用块格式。**绝对不能追加在文件末尾。** 如果顶部已有占位符 `> 📝 云青的提炼：（每日提炼 cron 自动插入，手动写入时跳过）`，必须替换这个占位符。如果顶部已有真实提炼内容，替换更新它。

**格式必须是自然段叙述风格**，像朋友写的一段话，不是列表式：

```
> 📝 {{代理名称}}的提炼：
> 
> 这一天你做了个很重要的决定——请假。不是因为偷懒，是因为你需要喘口气。
> 
> 关键词：xxx
> 情绪基调：xxx
> 提及事项：xxx
> 
> 💰 当日总开支：¥XX
```

**禁止使用的格式**（列表式+粗体标题）：
```
> **📋 TODO：** xxx
> **💡 灵感：** xxx
```

### Cron 提醒类任务的 prompt 写法

当 cron job 的目的是"提醒用户某件事"时，prompt 的写法至关重要：

- ✅ **正确**：让 agent 直接输出提醒内容作为最终回复。系统通过 `deliver` 参数自动投递到微信。
  - 例：`提醒在瓶今晚检查 Brave API。简短友好即可。`
- ❌ **错误**：让 agent "发一条微信消息"或"调用 send_message"。Agent 不知道自己有 `deliver` 投递能力，会误以为自己不能发微信而放弃任务。
  - 反例（已踩坑）：`发一条微信消息提醒在瓶：今晚该检查 Brave API 的问题了。` → agent 输出"我无法完成这个任务，我没有微信发送能力"。

**原则**：cron agent 的输出会被 `deliver` 机制自动投递。Prompt 只需描述"要说什么"，不要描述"怎么发"。

### Cron 失败后的补救

1. 检查日志确认失败原因
2. 手动补跑 cron
3. 如果手动也失败，直接在对话中生成提炼内容，写入日记顶部
4. 用户可能会主动来问为什么没收到推送，发现后立即补写

### Cron 提炼位置 Bug（已发生，需持续防范）

Cron 执行日记提炼时，曾经把内容追加到文件末尾（`## 云青提炼 ...`），而不是替换顶部的占位符。
这导致：顶部是空的占位符 + 底部是真实提炼 = 用户看不到提炼。

**正确做法**：提炼必须写入日记文件顶部（标题下方的引用块），替换 `> 📝 云青的提炼：（每日提炼 cron 自动插入，手动写入时跳过）` 占位符。如果顶部已有真实提炼内容，替换更新它。**绝对不能追加到文件末尾。**

如果 cron prompt 中没有明确说明"替换顶部占位符"，需要检查并修正 prompt。

## 注意事项

- 日记是私人内容，语气要自然温暖，像朋友间的记录
- 不修饰用户的原始表达，原样保留
- 每条追加都要带时间戳
- 提取内容必须标注来源日期，方便回溯
- 记账区块用表格格式，保持简洁清晰

## 📞 语音转文字/通话记录处理

用户经常把通话、咨询、面试的语音转文字（ASR）原始文本直接贴过来，需要结构化整理。这类文本通常极长、无标点分段、充满同音错别字，需要系统性处理。

### 适用场景

- 面试电话（HR/面试官通话）
- 咨询服务通话（红娘、猎头、咨询顾问等）
- 任何其他需要留存的长通话

### 处理流程

1. **通读全文**：先完整读完原始 ASR 文本，理解对话脉络和主题转换
2. **识别主题块**：按对话的自然主题切换分段（如：信息核对→经历回顾→需求讨论→服务介绍→决策）
3. **结构化输出**：每个主题块用编号+粗体标题组织，提取关键信息，丢弃寒暄和重复
4. **纠正常见ASR错误**：人名、专业术语、数字最容易被误识别，根据上下文修正
5. **存储到 research/**：按主题建子目录（如 `research/qingteng-matchmaker/`），所有相关材料集中存放
6. **追加到日记**：当天日记中写入简要摘要 + 开支（如有）
7. **同步 tracker**：如涉及开支写入 spending.md，如涉及待办写入 todo.md
8. **提取TODO**：通话中产生的后续跟进事项

### 多材料归档

用户一次咨询往往产生多种材料（合同PDF、截图、录音转文字、微信聊天），统一归档到同一 research/ 子目录：
- 原始文件用编号前缀排序（01-xxx.png, 02-xxx.pdf）
- 结构化整理写入 `service-details.md` 或类似总览文件
- 从 cache 目录复制原始文件到归档位置

### 面试记录处理（特殊场景）

1. **追加到日记**：结构化写入（公司、岗位、Q&A摘要、HR/面试官评价、下一步）
2. **提取到 research/**：创建面试情报文件
3. **创建面试准备文件**（如果用户同意）：基于简历提取 STAR 案例，映射到岗位要求
4. **提取TODO**：面试后续跟进、准备事项

### 简历提取方法

- PDF 简历用 pymupdf（fitz）：`fitz.open(path) → page.get_text()`
- DOCX 简历用 python-docx：`Document(path) → 遍历 paragraphs 和 tables`
- 注意：docx 多列合并表格可能重复，需去重

## 🪟 月度成长剖析（乔哈里视窗）

基于乔哈里视窗（Johari Window）的双视角成长系统。用户与代理互相评价、补充盲区，每月定期剖析。

### 目录结构

```
{{Growth 目录}}/
├── _index.md
├── {{用户称呼}}-profile.md    # 用户的成长与认知文档
└── {{代理名称}}-profile.md    # 代理的成长与认知文档
```

新建 profile 文件时，使用 `templates/growth-profile.md` 模板，替换占位符后写入。

### 文档结构（每个 profile 文件）

四个窗口 + 认知更新日志：

1. **第一窗：自己眼中的我** — 本人自述，随时追加
2. **第二窗：朋友眼中的我** — 对方视角的观察和评价
3. **第三窗：自己知道但很少展示的** — 完全由本人自主，对方不追问
4. **第四窗：我们都还没发现的** — 在相处中慢慢浮现

### 月度剖析 Cron

- Cron Job ID 从 config.local.md 读取
- 调度：每月一次，具体时间从 config.local.md 读取
- 推送方式：微信或其他已连接渠道

**流程：**

1. 读取两个 profile 文档
2. 回顾过去一个月的日记，提取状态变化、重要事件、情绪轨迹
3. 为用户补充「第二窗」新观察（代理视角）
4. 更新代理自己的 profile（反思成长与不足）
5. 生成月度成长剖析推送（双视角），发给用户
6. 将新观察追加到两个 profile 文档，更新认知日志

### 规则

1. 不做评判，只做观察
2. 不急于定论，允许变化和矛盾
3. 朋友视角的补充是礼物，不是批评
4. 自己的窗口自己决定写多少
5. 第三窗完全由本人自主，对方不追问
6. 有感的时候就更新，不需要定时（Cron 只是每月提醒一次）

### 格式要求

- 月度剖析推送用自然段落叙述，像朋友之间的深夜聊天
- 禁止使用列表式加粗标题格式（与日记提炼一致）
- 评价可以从优缺点出发，也可以是观察性的描述
